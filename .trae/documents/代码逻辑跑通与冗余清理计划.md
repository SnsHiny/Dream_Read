## 目标
对当前 Dream 项目进行一次端到端的“能跑通每个功能点”检查，并识别/归纳代码中的臃肿与冗余点，输出可执行的修复与精简方案（不在本阶段直接改业务代码，待你确认计划后再进入执行）。

## 范围
- 前端：路由与鉴权、登录/注册、信息采集(onboarding)、解梦、梦境记录(History)、画像(Profile)、退出登录、错误与加载态
- 后端：鉴权接口、用户写入/读取、梦境写入/读取/删除、画像生成/刷新/趋势、AI 调用链路、Supabase 交互与降级（memory store）
- 数据库：users/dreams/profiles 三表字段映射与数据一致性（snake_case ↔ camelCase）、时间字段、分页与搜索

## 交付物
1) 功能跑通检查报告（按用户路径分段，列出：入口、请求链路、关键状态、预期/实际、风险点、修复建议）
2) 冗余与臃肿清单（重复逻辑、重复映射、重复 UI/状态、临时兼容代码、可抽象/可删除项）
3) 精简与一致性改造方案（最小改动优先，含改动文件清单与验证方法）

## 检查方法（执行阶段会严格按此跑通）
### A. 入口与路由（首次访问/按钮跳转）
1. 入口 `/`、按钮“开始探索/查看记录”点击行为：未登录 → `/login`；登录成功 → 判断是否首登（nickname 为空）→ `/onboarding`；否则进入目标页。
2. 路由守卫：`/dream` `/history` `/profile` 等页面在无 user 状态下的跳转目标（是否应统一到 `/login` 而不是 `/onboarding`）。
3. 退出登录：清理 store + 本地持久化 + 页面回到 `/login`，并验证各页面不会因残留状态“误认为登录”。

### B. 登录链路（手机号验证码）
1. 前端登录页：验证码发送/校验、登录成功后 store.user 的字段形态（id/_id/phoneNumber/nickname 等）。
2. 后端登录路由：是否保证对前端返回统一 camelCase user；是否存在“仅写入 phone_number 但其他字段读取/更新异常”的分支。

### C. 信息采集（onboarding）→ 用户表写入
1. 前端提交 payload 与后端 `createOrUpdateUser` 的字段对应关系（camelCase → snake_case）。
2. Supabase 写入失败时的 memory store 降级：前端是否能感知并提示“数据不会持久化”；重启后是否丢失导致用户体验问题（作为风险项列出）。

### D. 解梦（AI 分析）→ dreams 表写入
1. 前端：创建梦境请求是否携带正确 userId（统一使用 user.id；兼容 user._id 但逐步收敛）。
2. 后端：AI 调用（火山引擎）失败/超时、重试策略、超时上限、错误返回码；写入 dreams 表字段是否与 DB 一致（user_id/content/input_type/tags/analysis/created_at）。
3. UI：等待态是否为全屏沉浸式；取消/返回/多次提交防抖；错误态是否可恢复。

### E. 记录页（History）→ dreams 表读取
1. 列表请求：分页/搜索参数是否与后端一致；返回结构是否被 store 正确解析（success/dreams/pagination）。
2. 时间显示：created_at/createdAt 的字段来源与格式化，时区问题与无效值兼容。
3. 列表渲染：key 唯一性、移动端点击区域覆盖、详情弹窗/抽屉交互。

### F. 画像页（Profile）→ profiles 表读取/生成/刷新
1. 画像数据来源梳理：首次无 profile 时由后端读取最近 50 条 dreams 生成画像；刷新接口是否 upsert 并返回一致结构。
2. “新增梦境后画像实时调整”：确认触发机制（前端在 createDream 成功后自动 refreshProfile / 或后端异步更新），并评估成本与体验。
3. 如果画像生成也依赖 AI：统一复用同一套“全屏沉浸等待动画”，并在 UI 上区分“解梦中/画像生成中”。
4. 饼图与整体风格一致性：当前图表组件是否可替换为更贴合的表达（如星环/雾化环形条、渐变与玻璃拟态 tooltip）。

## 冗余/臃肿检查维度（执行阶段会逐项标注）
1. **ID 字段冗余**：user.id vs user._id；dream.id vs dream._id；统一类型与映射层，减少分支判断。
2. **返回结构冗余**：API 层与 store 层重复做 response 兼容（response.user || response）；建议统一在 api.ts 归一化。
3. **映射冗余**：mapUserToDb/mapUserFromDb、mapProfileToDb/mapProfileFromDb 是否散落多处；建议集中到单一模块导出。
4. **降级冗余**：supabase 失败 fallback 到 memory store 的分支是否覆盖全面且行为一致；必要时加显式“当前为临时存储”标识。
5. **UI 冗余**：重复的 Loading/Overlay、重复的按钮样式与布局逻辑；抽象为复用组件。

## 验证策略（执行阶段）
1. 前端/后端分别启动 dev server，按上述 A-F 走完整用户旅程做 smoke test。
2. 在关键接口增加最小必要的日志（不记录隐私内容），确认每次点击对应的请求与响应结构。
3. 直接查询 Supabase 表验证：users/dreams/profiles 的写入是否发生、字段是否正确、created_at 是否存在。

## 预期改动方向（执行阶段才会动代码）
- 路由守卫与按钮跳转统一：未登录统一去 `/login`；登录后按首登逻辑分流。
- API 响应归一化：在前端 api 层统一做 id/字段映射，store 与页面不再写兼容分支。
- dreams 列表读取修复：确保 History 必然用 user.id 请求，且对 created_at 映射一致。
- 画像实时更新：createDream 成功后触发 refreshProfile（可做节流/延迟），并在画像刷新期间展示同款沉浸等待动画。
- UI：沉浸等待动画组件抽象为全局 Overlay，可在“解梦/画像生成”复用；退出按钮做成玻璃拟态的图标按钮（PC 顶部、移动端放在 Profile 页为主）。

## 需要你确认的点（确认后进入执行）
1) 未登录访问受限页时：统一跳转 `/login`（而非 `/onboarding`）是否符合你的产品逻辑？
2) “新增梦境后画像实时更新”期望：每次都立即刷新画像，还是仅在用户打开画像页时刷新？
